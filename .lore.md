## Long-term Knowledge

### Architecture

<!-- lore:019eb222-336e-78db-aeb7-2c400c4d971f -->
* **@cloudflare/sandbox@0.12.0 key SDK internals and container lifecycle**: \`@cloudflare/sandbox@0.12.0\` key internals: \`keepAlive:true\` means container never auto-timeouts — must call \`sandbox.destroy()\` explicitly. \`sleepAfter\` ignored when \`keepAlive:true\`; default is 10m — agent runs need \`sleepAfter: '2h'\` (set in \`getSandbox\` in \`webhooks/index.ts\`). \`SANDBOX\_TRANSPORT\`: 'http'|'websocket'|'rpc'; warns+defaults to 'http' if invalid. \`tunnels\` getter requires RPC transport. \`GET /v1/sandbox/{id}/running\` always returns HTTP 200; inspect \`running\` field. \`getContainerPlacementId()\` returns \`null\`/\`undefined\` until after \`exec()\`. ESM scripts in containers must be co-located with \`node\_modules\` — symlink or copy scripts into the SDK directory (e.g. \`/opt/opencode-sdk/\`) for correct import resolution.

<!-- lore:019eb222-3372-7538-b5e9-eb9ee82d1e13 -->
* **Container-to-Worker communication: no internal Cloudflare network path available**: No direct internal Cloudflare network path exists for container→Worker communication in \`@cloudflare/sandbox@0.12.0\` or \`@cloudflare/containers@0.3.7\`. The \`controlCallback\` (\`SandboxControlCallbackImpl\`) is \`private readonly\` on the \`Sandbox\` class — inaccessible to user code. Internal hostnames (\`r2.internal\`, \`s3-credential-proxy.internal\`) are mount-system internals only. The RPC transport's \`ContainerControlClient\` is for sandbox SDK internal use. Containers must call the public Worker URL (e.g. \`$WORKER\_URL/api/sessions/save\`) for callbacks — there is no shortcut through the Cloudflare network.

Container dispatches to root OpenCode session (\`parentID\` absent) with \`{ agent: 'jared', prompt: { text } }\`. Use root-session lookup over \`sessions\[0]\` (child sessions can sort first). Session-list curl in \`dispatch.ts\` line 127 has no \`--max-time\` flag — acceptable because readiness check confirms \`/session\` responds in ~2s; do not add speculative timeouts before observing deployed behavior.

### Gotcha

<!-- lore:019eb2ed-4372-7dc6-8b9c-0f8d17d2e1ec -->
* **Outpost webhook: pending events are never auto-retried after dispatch failures**: Outpost webhook: pending events are never auto-retried after dispatch failures. A pending row with \`dispatched\_at=null\` is not proof of success — no retry path exists. CRITICAL cold-start trap: destroying a container and retriggering causes the full cold-start sequence (boot + clone + serve + wait-for-OpenCode) to exceed the \`waitUntil\` budget, so \`dispatchPrompt\` never runs → event stays \`pending\`. On a warm container, the fast-path in \`ensureSandboxReady\` (line 37 of dispatch.ts) returns immediately if OpenCode is already running, and dispatch completes successfully → event reaches \`dispatched\`. This is confirmed behavior. Additional traps: deploying a new Worker version resets the Durable Object, killing in-progress \`waitUntil\`. \`withTimeout\` wrapping \`ensureSandboxReady\` with \`setTimeout\` inside \`waitUntil\` may not fire reliably after response is sent — remove it entirely. Debugging: \`wrangler tail\` sampling can drop events — use D1 records and container logs as ground truth.

<!-- lore:019ecc5c-75f8-7b73-8592-d4045a56e74b -->
* **Tailwind flexbox overflow: missing min-w-0 on flex children causes page-wide overflow**: Trap: a flex child with \`-m-6\` full-bleed negative margin looks like it only affects spacing, but without \`min-w-0\` on the flex item, \`min-width: auto\` allows wide content (buttons, headers) to expand the item beyond the available viewport width — especially visible next to a fixed nav sidebar. Fix: add \`min-w-0\` to the \`SidebarInset\` content wrapper in \`layout.tsx\` AND to the root div and header row in \`container-detail.tsx\`; add \`shrink-0\` to action button groups so they don't compress. Inner scroll regions with \`whitespace-pre-wrap break-words\` and \`break-all\` are unaffected.

### Preference

<!-- lore:019ecbfd-cda0-7ff2-9ee9-406c93be143a -->
* **Always destroy and retrigger the container after deploying a fix to validate it end-to-end**: After deploying a fix to the Cloudflare Worker, the user consistently destroys the container for the target entity (e.g., \`getsentry/cli#1099\`) and retrigggers/relabels it to create a fresh dispatch cycle. This is the standard validation step: deploy → destroy container → wait ~75-90s → check D1/session data to confirm the fix worked. The assistant should proactively suggest or execute this destroy-and-retrigger step immediately after every successful deploy, and then poll D1 or the detail endpoint after the wait period to report whether sessions were created and prompts were dispatched.

<!-- lore:019ec403-ef05-7817-801d-aaa3d38537a5 -->
* **Always destroy container and relabel to recover stuck/stale sessions**: When a container or session is stuck (no LLM activity, empty messages, stale data), the user's consistent recovery pattern is: (1) destroy/delete the stale container or D1 session record, then (2) remove and re-add the GitHub label to trigger a fresh webhook dispatch. The user expects the assistant to guide this exact sequence after diagnosing the root cause. If container destroy requires auth (dashboard-only), the assistant should fall back to D1 DELETE of the session row, then instruct the user to relabel from the UI. This pattern applies whenever session\_data shows 174 bytes (stub only), cost=0, tokens=0, or messages=\[].

<!-- lore:019ebf3e-7205-7e5b-82d9-99a99792b05f -->
* **Always extend trigger mechanisms to new platforms by mirroring existing GitHub patterns**: Always extend trigger mechanisms to new platforms by mirroring existing GitHub patterns: named trigger, full context fetch, same container/dispatch path, and a callback back to the source platform. Sentry follows this with HMAC-SHA256 verification via \`SENTRY\_INTEGRATION\_CLIENT\_SECRET\` and issue context via \`SENTRY\_INTEGRATION\_TOKEN\`. Prefer lightweight comment/label-based triggers (e.g., 'jared' label, \`/jared\` comment prefix) over formal account/OAuth setups — easy to invoke manually, no special provisioning needed.

When reading multiple files in a session, deliver complete contents for each one. Never abbreviate, skip sections, or replace code with descriptions like '// ... rest of file'.
