## Long-term Knowledge

### Architecture

<!-- lore:019eb222-336e-78db-aeb7-2c400c4d971f -->
* **@cloudflare/sandbox@0.12.0 key SDK internals and container lifecycle**: \`Sandbox\` DO constructor reads \`SANDBOX\_TRANSPORT\` ('http'|'websocket'|'rpc'; warns+defaults to 'http' if invalid). \`keepAlive:true\` means container never auto-timeouts — must call \`sandbox.destroy()\` explicitly or leak resources. \`sleepAfter\` ignored when \`keepAlive:true\`. Default \`sleepAfter\` is 10m — agent runs (15-30 min) need \`sleepAfter: '2h'\` to avoid mid-work container kills (set in \`getSandbox\` call in \`webhooks/index.ts\`). \`getContainerPlacementId()\` returns \`null\` (no env var) or \`undefined\` (no handshake yet); call \`exec()\` first. \`tunnels\` getter requires RPC transport. \`GET /v1/sandbox/{id}/running\` always returns HTTP 200; inspect \`running\` field. \`WriteResponse.ok\` and \`OkResponse.ok\` are always \`true\` on success. Keepalive status checked via \`pgrep -f 'keepalive.sh'\` in debug endpoint. ESM scripts in containers must be co-located with their \`node\_modules\` — import resolution uses script location, not cwd. Symlink or copy scripts into the SDK directory (e.g. \`/opt/opencode-sdk/\`) to resolve imports correctly.

<!-- lore:019eb222-3372-7538-b5e9-eb9ee82d1e13 -->
* **Container-to-Worker communication: no internal Cloudflare network path available**: No direct internal Cloudflare network path exists for container→Worker communication in \`@cloudflare/sandbox@0.12.0\` or \`@cloudflare/containers@0.3.7\`. The \`controlCallback\` (\`SandboxControlCallbackImpl\`) is \`private readonly\` on the \`Sandbox\` class — inaccessible to user code. Internal hostnames (\`r2.internal\`, \`s3-credential-proxy.internal\`) are mount-system internals only. The RPC transport's \`ContainerControlClient\` is for sandbox SDK internal use. Containers must call the public Worker URL (e.g. \`$WORKER\_URL/api/sessions/save\`) for callbacks — there is no shortcut through the Cloudflare network.

<!-- lore:019eb6d2-43d7-7d24-9208-dbb59c128e46 -->
* **Outpost client routing and nav structure**: Outpost client routing/nav: React Router routes in \`App.tsx\` are \`/login\`, \`/\`, \`/events\`, \`/events/:id\`, \`/containers\`, and \`/containers/:entityKey\`; all non-login routes sit under \`AuthGuard > Layout\`. Chose Hono RPC client paths to stay typed over ad hoc fetches because renaming routes breaks the generated client; \`getSessionDetail()\` is the deliberate raw-fetch exception.

<!-- lore:019eb6d2-43d6-7a33-9411-6df2656da3d4 -->
* **Outpost containers/sessions data flow: container→internal→D1→UI**: Outpost container dispatch must send the current OpenCode prompt shape: \`dispatchPrompt()\` targets the root session (\`parentID\` absent) and posts \`{ agent: 'jared', prompt: { text } }\`. Trap: \`{ parts: \[...] }\` looks plausible because message payloads elsewhere use parts, but this endpoint ignores it, so webhook work appears to succeed while the agent never receives a usable prompt.

<!-- lore:019eb6d2-43d6-7a33-9411-6df3ce9eb242 -->
* **Outpost DB schema: agentSessions table is the container session store**: Outpost container dispatch must target the root OpenCode session (\`parentID\` absent) and send \`{ agent: 'jared', prompt: { text } }\`. Chose root-session lookup over \`sessions\[0]\` because child sessions can sort first, and chose \`prompt.text\` over \`{ parts: \[...] }\` because this endpoint ignores parts and silently drops the dispatch.

<!-- lore:019eb214-34b7-7481-b8ce-463058803d05 -->
* **Outpost server auth middleware: two-layer pattern**: Outpost auth middleware is intentionally two-layered: global \`auth()\` always runs and only populates \`c.user\`/\`c.session\`, while \`isAuthenticated()\` is the actual 401 gate on selected routes. Chose this over making global auth reject requests because unauthenticated webhook/container routes still need to pass through with context attached.

<!-- lore:019eb214-34b8-7228-83bf-f1d37a9aae6a -->
* **Outpost sessions router: intentional unauthenticated /save route**: Outpost webhooks router: \`src/routes/webhooks/index.ts\` mounts \`.route('/github-app', githubRouter).route('/sentry', sentryRouter)\`. GitHub logic lives in \`src/routes/webhooks/github.ts\`; Sentry handler in \`src/routes/webhooks/sentry.ts\`; shared dispatch logic in \`src/lib/containers/dispatch.ts\`. Sessions router: \`POST /sessions\` is intentionally unauthenticated because it is called from inside containers; \`isAuthenticated()\` is applied after that route. \`OPENCODE\_PORT = 4096\` is defined in \`dispatch.ts\` and imported by \`containers/index.ts\`—do not redefine it locally.

<!-- lore:019eb2ed-4374-7691-8135-4aca74a1312c -->
* **Outpost webhook: pull\_request\_review events require app installation before review is posted**: GitHub App must be installed on a repo BEFORE a review is posted to receive \`pull\_request\_review\` events for that review. Events are not backfilled. For \`pull\_request\_review.submitted\` on bot-authored PRs: \`pull\_request.user.login = jared-outpost\[bot]\` → \`isBotEntity = true\` via existing \`prAuthor\` check — no special handling needed. \`PR\_EVENTS\` set in \`apps/server/src/lib/github/entity.ts\` includes \`pull\_request\_review\` and \`pull\_request\_review\_comment\`.

### Gotcha

<!-- lore:019eba46-c690-79fb-9fa2-4a7d1fe39862 -->
* **dispatchPrompt must target root session (no parentID), not sessions\[0]**: Trap: \`sessions\[0]\` looks right because it is the first returned session, but child sessions can sort ahead of the root. Fix: choose the session with no \`parentID\`, because only the root OpenCode session accepts dispatch prompts; targeting a child session silently misroutes webhook work.

<!-- lore:019eb2ed-4372-7dc6-8b9c-0f8d17d2e1ec -->
* **Outpost webhook: pending events are never auto-retried after dispatch failures**: Outpost webhook: pending events are never auto-retried after dispatch failures. A pending row is not proof of success — there is no retry path. Real failures are usually downstream in sandbox startup/readiness or prompt dispatch. Fix: capture live Worker logs during a fresh relabel and trace the exact stage before \`saveInitialSession()\` instead of inferring from stale DB rows. Silent failures inside \`waitUntil\` mean a missing \`failed\` row also does not confirm success.

### Preference

<!-- lore:019eba7e-a110-744b-a852-68e5ff4108d1 -->
* **Always destroy and retrigger stuck containers rather than waiting or debugging in place**: Always destroy and retrigger stuck containers rather than waiting or debugging in place: when a container or agent session appears stuck, idle, or producing no activity, the user's consistent response is to destroy the container and retrigger it (for example by removing and re-adding the \`jared\` label). This is preferred over in-place recovery because a clean slate is faster and avoids chasing stale state; do a quick diagnostic first only to confirm the stuck state.

<!-- lore:019ebf3e-7205-7e5b-82d9-99a99792b05f -->
* **Always extend trigger mechanisms to new platforms by mirroring existing GitHub patterns**: Always extend trigger mechanisms to new platforms by mirroring existing GitHub patterns: named trigger, full context fetch, same container/dispatch path, and a callback back to the source platform. Sentry follows this with HMAC-SHA256 verification via \`SENTRY\_INTEGRATION\_CLIENT\_SECRET\` and issue context via \`SENTRY\_INTEGRATION\_TOKEN\`. Prefer lightweight comment/label-based triggers (e.g., 'jared' label, \`/jared\` comment prefix) over formal account/OAuth setups — easy to invoke manually, no special provisioning needed.
