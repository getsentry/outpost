## Long-term Knowledge

### Architecture

<!-- lore:019eb222-336e-78db-aeb7-2c400c4d971f -->
* **@cloudflare/sandbox@0.12.0 key SDK internals and container lifecycle**: \`@cloudflare/sandbox@0.12.0\` key internals: \`keepAlive:true\` means container never auto-timeouts — must call \`sandbox.destroy()\` explicitly. \`sleepAfter\` ignored when \`keepAlive:true\`; default is 10m — agent runs need \`sleepAfter: '2h'\` (set in \`getSandbox\` in \`webhooks/index.ts\`). \`SANDBOX\_TRANSPORT\`: 'http'|'websocket'|'rpc'; warns+defaults to 'http' if invalid. \`tunnels\` getter requires RPC transport. \`GET /v1/sandbox/{id}/running\` always returns HTTP 200; inspect \`running\` field. \`getContainerPlacementId()\` returns \`null\`/\`undefined\` until after \`exec()\`. ESM scripts in containers must be co-located with \`node\_modules\` — symlink or copy scripts into the SDK directory (e.g. \`/opt/opencode-sdk/\`) for correct import resolution.

<!-- lore:019eb222-3372-7538-b5e9-eb9ee82d1e13 -->
* **Container-to-Worker communication: no internal Cloudflare network path available**: No direct internal Cloudflare network path exists for container→Worker communication in \`@cloudflare/sandbox@0.12.0\` or \`@cloudflare/containers@0.3.7\`. The \`controlCallback\` (\`SandboxControlCallbackImpl\`) is \`private readonly\` on the \`Sandbox\` class — inaccessible to user code. Internal hostnames (\`r2.internal\`, \`s3-credential-proxy.internal\`) are mount-system internals only. The RPC transport's \`ContainerControlClient\` is for sandbox SDK internal use. Containers must call the public Worker URL (e.g. \`$WORKER\_URL/api/sessions/save\`) for callbacks — there is no shortcut through the Cloudflare network.

<!-- lore:019eb6d2-43d7-7d24-9208-dbb59c128e46 -->
* **Outpost client routing and nav structure**: Outpost client routing/nav: React Router routes in \`App.tsx\` are \`/login\`, \`/\`, \`/events\`, \`/events/:id\`, \`/containers\`, and \`/containers/:entityKey\`; all non-login routes sit under \`AuthGuard > Layout\`. Chose Hono RPC client paths to stay typed over ad hoc fetches because renaming routes breaks the generated client; \`getSessionDetail()\` is the deliberate raw-fetch exception.

<!-- lore:019eb6d2-43d6-7a33-9411-6df2656da3d4 -->
* **Outpost containers/sessions data flow: container→internal→D1→UI**: Outpost container dispatch targets root OpenCode session (\`parentID\` absent) with \`{ agent: 'jared', prompt: { text } }\`. Use root-session lookup over \`sessions\[0]\` because child sessions can sort first. Use \`prompt.text\` over \`{ parts: \[...] }\` — endpoint silently drops parts. Normalize both bare-array and \`{ data: \[...] }\` responses via \`unwrapArray(raw: string): unknown\[]\` in \`containers/index.ts\` (NOT \`parseListResponse()\` which casts to \`Array<{id:string}>\` and breaks message shapes). Sessions still cast to \`Array<{id:string}>\` at call site; messages use \`unknown\[]\` preserving \`{info, parts}\` shape. Prompt fallback chain in \`dispatch.ts\`: \`/api/session/:id/prompt\` → \`/session/:id/prompt\` → \`/session/:id/prompt\_async\`. \`prompt\_async\` must NOT be tried first — returns 204 but silently drops prompt on OpenCode v1.17.0.

<!-- lore:019eb214-34b7-7481-b8ce-463058803d05 -->
* **Outpost server auth middleware: two-layer pattern**: Outpost auth middleware is intentionally two-layered: global \`auth()\` always runs and only populates \`c.user\`/\`c.session\`, while \`isAuthenticated()\` is the actual 401 gate on selected routes. Chose this over making global auth reject requests because unauthenticated webhook/container routes still need to pass through with context attached.

<!-- lore:019eb214-34b8-7228-83bf-f1d37a9aae6a -->
* **Outpost sessions router: intentional unauthenticated /save route**: Outpost webhooks router: \`src/routes/webhooks/index.ts\` mounts \`.route('/github-app', githubRouter).route('/sentry', sentryRouter)\`. GitHub logic lives in \`src/routes/webhooks/github.ts\`; Sentry handler in \`src/routes/webhooks/sentry.ts\`; shared dispatch logic in \`src/lib/containers/dispatch.ts\`. Sessions router: \`POST /sessions\` is intentionally unauthenticated because it is called from inside containers; \`isAuthenticated()\` is applied after that route. \`OPENCODE\_PORT = 4096\` is defined in \`dispatch.ts\` and imported by \`containers/index.ts\`—do not redefine it locally.

<!-- lore:019eb2ed-4374-7691-8135-4aca74a1312c -->
* **Outpost webhook: pull\_request\_review events require app installation before review is posted**: GitHub App must be installed on a repo BEFORE a review is posted to receive \`pull\_request\_review\` events for that review. Events are not backfilled. For \`pull\_request\_review.submitted\` on bot-authored PRs: \`pull\_request.user.login = jared-outpost\[bot]\` → \`isBotEntity = true\` via existing \`prAuthor\` check — no special handling needed. \`PR\_EVENTS\` set in \`apps/server/src/lib/github/entity.ts\` includes \`pull\_request\_review\` and \`pull\_request\_review\_comment\`.

<!-- lore:019ec325-ec1a-7fdb-ae7b-3282a8b014cf -->
* **Real-time STT pipeline: Soniox + ElevenLabs routing by language coverage**: Speech-to-text pipeline uses two providers routed by language: Soniox (~60 languages, $0.12/hr) handles its supported languages including English (most-used); ElevenLabs Scribe v2 realtime (~40 additional languages, $0.39/hr) handles all others. LLM post-processing uses Groq for fastest inference (high TPS/TTFT). Qwen3 32B preferred over Llama 3.1 8B despite ~200ms speed disadvantage because of superior multilingual quality (English + Spanish + Chinese benchmark). Llama 3.1 8B and GPT OSS 20B rejected — latter was 'surprisingly bad'; large models rejected for responding 'like an agent' instead of returning transcribed text. Target time-tested models with great TPS/TTFT, NOT latest reasoning models.

### Gotcha

<!-- lore:019eb2ed-4372-7dc6-8b9c-0f8d17d2e1ec -->
* **Outpost webhook: pending events are never auto-retried after dispatch failures**: Outpost webhook: pending events are never auto-retried after dispatch failures. A pending row with \`dispatched\_at=null\` is not proof of success — no retry path exists. CRITICAL: deploying a new Worker version resets the Durable Object, killing any in-progress \`waitUntil\` context (\`Error: Durable Object reset because its code was updated.\`). \`saveInitialSession\` runs synchronously before \`waitUntil\`, so the stub (len=146) is written but async dispatch is killed — leaving status=pending indefinitely. OpenCode takes 30-40s after port opens to finish loading; poll \`GET /session\` or \`GET /api/session\` up to 60×1s after port opens before dispatching. The \`waitUntil\` budget is NOT the primary concern — \`sandbox.exec()\` calls are RPC calls to the Durable Object which maintains its own execution context. The DO keeps the async chain alive; \`waitUntil\` just prevents GC of the promise chain.

### Preference

<!-- lore:019ec403-ef05-7817-801d-aaa3d38537a5 -->
* **Always destroy container and relabel to recover stuck/stale sessions**: When a container or session is stuck (no LLM activity, empty messages, stale data), the user's consistent recovery pattern is: (1) destroy/delete the stale container or D1 session record, then (2) remove and re-add the GitHub label to trigger a fresh webhook dispatch. The user expects the assistant to guide this exact sequence after diagnosing the root cause. If container destroy requires auth (dashboard-only), the assistant should fall back to D1 DELETE of the session row, then instruct the user to relabel from the UI. This pattern applies whenever session\_data shows 174 bytes (stub only), cost=0, tokens=0, or messages=\[].

<!-- lore:019ebf3e-7205-7e5b-82d9-99a99792b05f -->
* **Always extend trigger mechanisms to new platforms by mirroring existing GitHub patterns**: Always extend trigger mechanisms to new platforms by mirroring existing GitHub patterns: named trigger, full context fetch, same container/dispatch path, and a callback back to the source platform. Sentry follows this with HMAC-SHA256 verification via \`SENTRY\_INTEGRATION\_CLIENT\_SECRET\` and issue context via \`SENTRY\_INTEGRATION\_TOKEN\`. Prefer lightweight comment/label-based triggers (e.g., 'jared' label, \`/jared\` comment prefix) over formal account/OAuth setups — easy to invoke manually, no special provisioning needed.

<!-- lore:019ec4e8-3397-7830-b06c-0ca79592135c -->
* **Always instruct to not deploy until explicitly given the signal**: The user consistently controls when deployments happen and explicitly signals readiness before any deploy or push. In prior sessions, the user explicitly said 'do not push or deploy anything yet' to avoid resetting Durable Objects and killing in-progress state. The user observes, diagnoses, and approves fixes first, then separately gives the go-ahead to deploy. Never deploy, push, or trigger a wrangler deploy command unless the user has explicitly said it's okay to do so in the current session.

<!-- lore:019ec08c-28f5-75ce-bdee-bdc3e0b5a3d1 -->
* **Always investigate D1 database state and container logs together when diagnosing stuck/idle container behavior**: When a container appears stuck, idle, or not producing expected output, the user consistently cross-references two data sources simultaneously: (1) the D1 database record (checking sessionData fields like messages, sessionStatus, session\_data\_len, updated\_at, cost, tokens) and (2) raw container logs (OpenCode internal logs, keepalive logs, process output). The user looks for discrepancies between what logs show (activity) and what D1 shows (stale/empty data), treating this gap as the diagnostic signal. The user expects the assistant to help interpret both sources together and identify the root cause (e.g., keepalive POST failing, messages array empty despite log activity).

<!-- lore:019eba2c-4f99-7337-9222-2ab57eff38be -->
* **Always investigate recent git commits when diagnosing regressions**: When something regresses, inspect the last 3-4 git commits first and diff the files related to the symptom. The user assumes recent changes are the most likely cause because issues often appear shortly after a working state, so commit history is part of the standard diagnosis path.

<!-- lore:019ecbbe-a101-71d5-8b26-96c9245dc130 -->

They expect analysis that pinpoints exact timestamps, identifies which step exceeded a budget or threshold, and explains the downstream consequence (e.g., why subsequent steps never executed). The user wants the full causal chain: what took too long → what budget/limit was exceeded → what failed to run as a result → why the observable symptom (no messages/sessions/logs) occurred. Always provide specific durations, timestamps, and log evidence when diagnosing failures.

* **Always typecheck, build, commit, push, and deploy as a single atomic sequence after code changes**: After every code fix, the user consistently follows this exact sequence: (1) run typecheck, (2) run build, (3) commit with a conventional commit message starting with 'fix:' describing the specific change, (4) push directly to main (bypassing CI/PR requirements), and (5) deploy via \`wrangler deploy\`. The user also expects a structured summary of changed files, routes/exports, and UI fields, because partial verification has missed regressions before.

The user prefers confirming whether the trigger fired, dispatched, and persisted, because pending rows and silent \`waitUntil\` failures can hide the real break point.
