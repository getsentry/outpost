### Architecture

* **@cloudflare/sandbox@0.12.0 key SDK internals and container lifecycle**: \`Sandbox\` DO constructor reads \`SANDBOX\_TRANSPORT\` ('http'|'websocket'|'rpc'; warns+defaults to 'http' if invalid). \`keepAlive:true\` means container never auto-timeouts — must call \`sandbox.destroy()\` explicitly or leak resources. \`sleepAfter\` ignored when \`keepAlive:true\`. Default \`sleepAfter\` is 10m — agent runs (15-30 min) need \`sleepAfter: '2h'\` to avoid mid-work container kills (set in \`getSandbox\` call in \`webhooks/index.ts\`). \`getContainerPlacementId()\` returns \`null\` (no env var) or \`undefined\` (no handshake yet); call \`exec()\` first. \`tunnels\` getter requires RPC transport. \`GET /v1/sandbox/{id}/running\` always returns HTTP 200; inspect \`running\` field. \`WriteResponse.ok\` and \`OkResponse.ok\` are always \`true\` on success. Keepalive status checked via \`pgrep -f 'keepalive.sh'\` in debug endpoint. ESM scripts in containers must be co-located with their \`node\_modules\` — import resolution uses script location, not cwd. Symlink or copy scripts into the SDK directory (e.g. \`/opt/opencode-sdk/\`) to resolve imports correctly.

* **Container-to-Worker communication: no internal Cloudflare network path available**: No direct internal Cloudflare network path exists for container→Worker communication in \`@cloudflare/sandbox@0.12.0\` or \`@cloudflare/containers@0.3.7\`. ### Gotcha

<!-- lore:019eba46-c690-79fb-9fa2-4a7d1fe39862 -->
* **dispatchPrompt must target root session (no parentID), not sessions\[0]**: Trap: \`dispatchPrompt\` picks \`sessions\[0]\` from OpenCode's \`/session\` list, which looks correct because there's usually one session. But OpenCode spawns subagent sessions (explore, task) with a \`parentID\` field — these appear in the list and can sort before the root session. Fix: filter sessions to find the one with no \`parentID\` (the root jared session). Sending a webhook event prompt to a subagent session causes it to be silently ignored or mishandled — the subagent has restricted permissions (no todowrite, no task) and doesn't know how to handle webhook events. File: \`apps/server/src/routes/webhooks/index.ts\` line 188.

<!-- lore:019eb22c-b0b6-7aa0-bd31-c8ad588c57eb -->
* **Outpost webhook: isBotEntity misses check\_suite/workflow\_run sender**: Trap: for \`workflow\_run\` events, \`sender.login\` is the human who triggered the workflow run (e.g. MathurAditya724), NOT the PR/commit author — so checking only \`sender.login\` against \`botLogin\` always returns false for bot-authored PRs triggered by humans. Fix: also check \`workflow\_run.head\_commit.author.name\` and \`workflow\_run.head\_commit.committer.name\`. Real payload confirmed: \`sender.login=MathurAditya724\` (no match), \`head\_commit.author.name=jared-outpost\[bot]\` (match). Applied in \`apps/server/src/routes/webhooks/index.ts\`.

<!-- lore:019eb2ed-4372-7dc6-8b9c-0f8d17d2e1ec -->
* **Outpost webhook: pending events are never auto-retried after dispatch failures**: Trap: events stuck as \`pending\` (e.g. from dispatch failures before a compatibility date fix) look like they'll eventually be processed — they won't. The \`retry-pending\` endpoint only marks them \`failed\`; it does NOT re-dispatch them. There is no automatic retry mechanism. After fixing the root cause of dispatch failures, stale pending events must be manually re-triggered (e.g. by re-adding the label on the PR) to get them dispatched.

<!-- lore:019eb22c-b0b8-7a34-862f-18a1dd17f3da -->
* **Outpost webhook: pull\_request.labeled dispatch silently fails via waitUntil**: Trap: dispatching events inside \`waitUntil\` swallows errors — event shows as \`pending\` in queue but is never dispatched, with no visible error. Debugging is hard because \`wrangler tail\` requires auth and Cloudflare runtime kills \`waitUntil\` context after response returns. Always add explicit error logging inside \`waitUntil\` callbacks. Confirmed root causes: (1) stale/incorrect \`compatibility\_date\` in \`wrangler.jsonc\` causes silent failures — events stay \`pending\` with no error; updating compat date resolved it. (2) Missing \`ContainerProxy\` export causes \`ctx.exports is undefined\` error in Sandbox DO alarm, surfaced only via \`wrangler tail\`. Fix: update compat date to \`2025-12-01\`+ and ensure \`ContainerProxy\` is exported from Worker entrypoint.
