## Long-term Knowledge

### Architecture

<!-- lore:019eb222-3370-79ff-a5e3-b0e167e28254 -->
* **@cloudflare/containers@0.3.7 outbound interception and container state machine**: Container state machine: stopped→running→healthy→stopping→stopped (also stopped\_with\_code). State stored in DO storage under \`\_\_CF\_CONTAINER\_STATE\`. Outbound interception priority (first match wins): 1. deniedHosts (520), 2. allowedHosts gate (520), 3. runtime per-host, 4. static per-host, 5. runtime catch-all, 6. static catch-all, 7. allowedHosts internet fallback, 8. enableInternet fallback, 9. default deny (520). \`interceptAll\` is permanent once promoted until instance restart. \`containerFetch()\` does NOT support WebSockets — use \`fetch()\` instead. \`startInFlight\` coalesces concurrent start calls to prevent double-start race. Must export \`ContainerProxy\` from Worker entrypoint for outbound interception to work.

<!-- lore:019eb222-3373-72d8-9076-a2034598b763 -->
* **@cloudflare/sandbox bridge API routes and auth**: All bridge routes under \`/v1\` prefix. Auth: Bearer token via \`SANDBOX\_API\_KEY\`; \`/openapi.\*\` also accepts \`?token=\`. Key routes: \`POST /v1/sandbox\` (create, returns base32 id), \`POST /v1/sandbox/{id}/exec\` (SSE stream: stdout/stderr base64 chunks, exit JSON), \`GET /v1/sandbox/{id}/running\` (always HTTP 200, check \`running\` field), \`DELETE /v1/sandbox/{id}\` (best-effort, unknown IDs return 204), \`POST /v1/sandbox/{id}/tunnel/{port}\` (port 3000 reserved, range 1024-65535). \`cwd\` in exec must resolve within /workspace (403 if not). File paths in read/write are without leading slash (e.g. \`workspace/main.py\`).

<!-- lore:019eb222-336e-78db-aeb7-2c400c4d971f -->
* **@cloudflare/sandbox@0.12.0 key SDK internals and container lifecycle**: \`Sandbox\` DO constructor reads \`SANDBOX\_TRANSPORT\` ('http'|'websocket'|'rpc'; warns+defaults to 'http' if invalid). \`keepAlive:true\` means container never auto-timeouts — must call \`sandbox.destroy()\` explicitly or leak resources. \`sleepAfter\` ignored when \`keepAlive:true\`. Default \`sleepAfter\` is 10m — agent runs (15-30 min) need \`sleepAfter: '2h'\` to avoid mid-work container kills (set in \`getSandbox\` call in \`webhooks/index.ts\`). \`getContainerPlacementId()\` returns \`null\` (no env var) or \`undefined\` (no handshake yet); call \`exec()\` first. \`tunnels\` getter requires RPC transport. \`GET /v1/sandbox/{id}/running\` always returns HTTP 200; inspect \`running\` field. \`WriteResponse.ok\` and \`OkResponse.ok\` are always \`true\` on success (enum constraint). Keepalive status checked via \`pgrep -f 'keepalive.sh'\` in debug endpoint.

<!-- lore:019eb222-3372-7538-b5e9-eb9ee82d1e13 -->
* **Container-to-Worker communication: no internal Cloudflare network path available**: No direct internal Cloudflare network path exists for container→Worker communication in \`@cloudflare/sandbox@0.12.0\` or \`@cloudflare/containers@0.3.7\`. The \`controlCallback\` (\`SandboxControlCallbackImpl\`) is \`private readonly\` on the \`Sandbox\` class — inaccessible to user code. Internal hostnames (\`r2.internal\`, \`s3-credential-proxy.internal\`) are mount-system internals only. The RPC transport's \`ContainerControlClient\` is for sandbox SDK internal use. Containers must call the public Worker URL (e.g. \`$WORKER\_URL/api/sessions/save\`) for callbacks — there is no shortcut through the Cloudflare network.

* **Outpost DB schema: agentSessions table is the container session store**: \`agentSessions\` table (Drizzle SQLite): \`entityKey\` PK (format \`owner/repo#number\`), \`sessionId\` (always null on insert), \`sessionData\` NOT NULL (JSON blob), \`createdAt\`, \`updatedAt\`. \`sessionData\` JSON structure: \`{sessionStatus: Record\<sessionId, {type: 'busy'|'idle'}>, sessions: Array<{id, title, cost, tokens, agent, model, parentID, ...}>, logs: string, messages: Record\<sessionId, messages\[]>}\`. Messages shape: \`Array<{info: {id, role, createdAt}, parts: \[{type, text?, toolName?, args?, result?, state?}]}>\`. \`webhookEvents\` table has composite index on \`(entityKey, status)\`. No separate messages table; messages embedded in \`sessionData\` JSON and merged/deduplicated in \`saveSession()\` by \`info.id\`.

### Gotcha

<!-- lore:019eba46-c690-79fb-9fa2-4a7d1fe39862 -->
* **dispatchPrompt must target root session (no parentID), not sessions\[0]**: Trap: \`dispatchPrompt\` picks \`sessions\[0]\` from OpenCode's \`/session\` list, which looks correct because there's usually one session. But OpenCode spawns subagent sessions (explore, task) with a \`parentID\` field — these appear in the list and can sort before the root session. Fix: filter sessions to find the one with no \`parentID\` (the root jared session). Sending a webhook event prompt to a subagent session causes it to be silently ignored or mishandled — the subagent has restricted permissions (no todowrite, no task) and doesn't know how to handle webhook events. File: \`apps/server/src/routes/webhooks/index.ts\` line 188.

<!-- lore:019eb22c-b0b6-7aa0-bd31-c8ad588c57eb -->
* **Outpost webhook: isBotEntity misses check\_suite/workflow\_run sender**: Trap: for \`workflow\_run\` events, \`sender.login\` is the human who triggered the workflow run (e.g. MathurAditya724), NOT the PR/commit author — so checking only \`sender.login\` against \`botLogin\` always returns false for bot-authored PRs triggered by humans. Fix: also check \`workflow\_run.head\_commit.author.name\` and \`workflow\_run.head\_commit.committer.name\`. Real payload confirmed: \`sender.login=MathurAditya724\` (no match), \`head\_commit.author.name=jared-outpost\[bot]\` (match). Applied in \`apps/server/src/routes/webhooks/index.ts\`.

<!-- lore:019eb2ed-4372-7dc6-8b9c-0f8d17d2e1ec -->
* **Outpost webhook: pending events are never auto-retried after dispatch failures**: Trap: events stuck as \`pending\` (e.g. from dispatch failures before a compatibility date fix) look like they'll eventually be processed — they won't. The \`retry-pending\` endpoint only marks them \`failed\`; it does NOT re-dispatch them. There is no automatic retry mechanism. After fixing the root cause of dispatch failures, stale pending events must be manually re-triggered (e.g. by re-adding the label on the PR) to get them dispatched.

<!-- lore:019eb22c-b0b8-7a34-862f-18a1dd17f3da -->
* **Outpost webhook: pull\_request.labeled dispatch silently fails via waitUntil**: Trap: dispatching events inside \`waitUntil\` swallows errors — event shows as \`pending\` in queue but is never dispatched, with no visible error. Debugging is hard because \`wrangler tail\` requires auth and Cloudflare runtime kills \`waitUntil\` context after response returns. Always add explicit error logging inside \`waitUntil\` callbacks. Confirmed root causes: (1) stale/incorrect \`compatibility\_date\` in \`wrangler.jsonc\` causes silent failures — events stay \`pending\` with no error; updating compat date resolved it. (2) Missing \`ContainerProxy\` export causes \`ctx.exports is undefined\` error in Sandbox DO alarm, surfaced only via \`wrangler tail\`. Fix: update compat date to \`2025-12-01\`+ and ensure \`ContainerProxy\` is exported from Worker entrypoint.
